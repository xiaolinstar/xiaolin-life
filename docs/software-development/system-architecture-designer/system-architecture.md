# 系统架构设计基础知识

## 系统架构的定义

概念：系统功能需求、非功能需求、质量属性，主要考虑系统的**非功能需求**。

体系结构=架构

## 设计与生命周期

需求分析-设计阶段-实现阶段-构件组装阶段-部署阶段-后开发阶段。

需求阶段，关注2个问题：
1. 如何根据**需求模型**构建软件架构模型
2. 如何保证模型转换的**可追踪性**

6个阶段：

1. 需求分析阶段
2. 设计阶段，架构模型的描述、软件架构模型的设计与分析方法，以及对软件架构设计经验的总结与复用等。 
3. 实现阶段，项目组织结构、变更配置管理；编码、测试；测试技术 
4. 构建组装阶段：失配问题。 
5. 部署阶段：部署图，部署方案 
6. 后开发阶段：维护、演化、复用

## 基于架构的软件开发方法

基于体系结构的软件设计（ABSD）方法，体系结构驱动，即**商业、质量和功能需求**组合驱动，采用**视角和视图**描述软件架构，用例描述功能需求，质量场景描述质量需求。

自顶向下，递归细化。

三个基础：
1. 功能分解
2. 体系结构风格
3. 软件模版的使用

划分为需求、设计、文档化、复审、实现、演化。

## 软件架构风格

某一特定应用领域中系统组织方式的惯用模式。一个体系结构定义一个**词汇表**和一组**约束**。

通用架构风格分类：
1. 数据流风格：批处理、管道过滤器，高内聚低耦合，支持并行。
2. 调用/返回风格：构件，主程序子程序、面向对象风格、层次型体系结构风格，客户端-服务器风格。
3. 仓库风格：数据中心，数据库系统，黑板系统风格（知识源、黑板数据结构、控制）
4. 虚拟机风格：解释器、规则系统
5. 独立架构风格：进程通信、事件驱动（隐式调用），放弃了对系统的控制
6. *闭环风格
7. C2风格：通过连接件绑定在一起的按照一组规则运作的并行构建网络。
