# 系统架构设计基础知识

## 系统架构的定义

概念：系统功能需求、非功能需求、质量属性，主要考虑系统的**非功能需求**。

体系结构=架构

## 设计与生命周期

需求分析-设计阶段-实现阶段-构件组装阶段-部署阶段-后开发阶段。

需求阶段，关注2个问题：
1. 如何根据**需求模型**构建软件架构模型
2. 如何保证模型转换的**可追踪性**

6个阶段：

1. 需求分析阶段
2. 设计阶段，架构模型的描述、软件架构模型的设计与分析方法，以及对软件架构设计经验的总结与复用等。 
3. 实现阶段，项目组织结构、变更配置管理；编码、测试；测试技术 
4. 构建组装阶段：失配问题。 
5. 部署阶段：部署图，部署方案 
6. 后开发阶段：维护、演化、复用

## 基于架构的软件开发方法

基于体系结构的软件设计（ABSD）方法，体系结构驱动，即**商业、质量和功能需求**组合驱动，采用**视角和视图**描述软件架构，用例描述功能需求，质量场景描述质量需求。

自顶向下，递归细化。

三个基础：
1. 功能分解
2. 体系结构风格
3. 软件模版的使用

划分为需求、设计、文档化、复审、实现、演化。

## 软件架构风格

某一特定应用领域中系统组织方式的惯用模式。一个体系结构定义一个**词汇表**和一组**约束**。

通用架构风格分类：
1. 数据流风格：批处理、管道过滤器，高内聚低耦合，支持并行。
2. 调用/返回风格：构件，主程序子程序、面向对象风格、层次型体系结构风格，客户端-服务器风格。
3. 仓库风格：数据中心，数据库系统，黑板系统风格（知识源、黑板数据结构、控制）
4. 虚拟机风格：解释器、规则系统
5. 独立架构风格：进程通信、事件驱动（隐式调用），放弃了对系统的控制
6. *闭环风格
7. C2风格：通过连接件绑定在一起的按照一组规则运作的并行构建网络。
8. 控制环路：将过程输出的指定属性维护在一个特定的参考值（设定值），将事物处理看成输入、加工、输出、反馈、再输入的一个持续过程模型。空调的自动调节器，巡航系统。

## 特定领域软件架构 DSSA

4个特点、3个基本活动、4种角色。

特定领域软件架构以一个特定问题领域为对象，形成由领域参考模型、参考需求、参考架构等组成的开发基础架构，其n标是支持一个特定领域中多个应用的生成。

DSSA 的基本活动包括领域分析、领域设计和领域实现。

1. 领域分析：获得领域模型，领域模型描述领域中系统之间的共同需求，即领域需求；
2. 领域设计：获得 DSSA，DSSA 描述领域模型中表示需求的解决方案；
3. 领域实现：依据领域需求和 DSSA 开发和组织可重用信息，并对基础软件架构进行实现。


垂直域、水平域

建立过程，并发的、递归的、反复的、螺旋的
1. 定义领域范围
2. 定义领域特定的元素
3. 定义领域特定的设计和实现需求约束
4. 定义领域模型和架构
5. 产生、搜集可重用的产品单元

领域工程、应用工程

DSSA 可复用构建应满足的质量属性：
- 可靠性：在指定条件下能稳定地完成预期功能，不易出错
- 通用性：能够适用于多种环境和场景，而非针对单一特定应用
- 可读性：代码已于他人理解和阅读，包括其功能和设计意图
- 可维护性：易于修改、更新和修复，以便适应需求变化和修复缺陷
- 可迁移性：方便地在不同的系统环境（如不同的操作系统、硬件平台）中部署和运行

## 系统架构评估

系统架构风险是指架构设计中潜在的、存在的问题的架构决策所带来的隐患。

敏感点是为了实现某种特定的质量属性，一个或多个系统构建所具有的特性。

权衡点是指影响多个质量属性，并对多个质量属性来说都是敏感点的系统属性。


风险承担者：利益相关人，不确定性。

场景：刺激、环境和响应

## 系统架构评估方法

三种：调查问卷或检查表、场景、度量

基于度量的评估方法：
1. SAAM：软件架构，可修改性、可移植性、可扩充性
2. ATAM：架构权衡，效用树（质量属性分类和优先级排序），案例分析一
3. CBAM：成本效益

**ATAM 质量属性效用树**，识别质量属性并排序，只有四个属性：
1. 性能
2. 可用性
3. 安全
4. 可修改性
